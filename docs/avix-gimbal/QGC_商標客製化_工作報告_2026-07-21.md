# QGC 商標客製化（White-Label）— 工作報告

日期：2026-07-21
範圍：QGroundControl v5.0.8，透過官方 `custom/` 客製化建置架構置換應用程式商標/圖示（系統圖示、桌面捷徑、應用程式內 logo）

---

## 1. 軟體拓樸

沒有新增任何獨立程式邏輯，完全沿用 QGC 官方既有的白牌客製化機制：

- 根目錄 `CMakeLists.txt` 在 configure 階段會檢查 `IS_DIRECTORY ${CMAKE_SOURCE_DIR}/custom`，只要這個目錄存在就自動開啟 `QGC_CUSTOM_BUILD`，不需要額外的建置參數或環境變數
- `custom/src/CustomPlugin`（繼承 `QGCCorePlugin`）取代預設的 `QGCCorePlugin::instance()`，覆蓋 `brandImageIndoor()`/`brandImageOutdoor()`
- `custom/src/CustomPlugin` 同時安裝一個 `QQmlAbstractUrlInterceptor`：攔截所有 `qrc:/res/*` 的查找，若 `:/Custom/res/*` 底下存在同名檔案，就自動改指過去。既有程式碼裡直接寫死的 `qrc:/res/QGCLogoFull.svg`、`:/res/qgroundcontrol.ico` 等引用完全不用改一行，執行期自動轉向客製化版本

**關鍵設計決策：**

| 項目 | 決策 | 原因 |
|---|---|---|
| 客製化機制 | 官方 `custom/` 白牌架構（複製 `custom-example/` 精簡而來），非自建機制 | 這是 QGC 原生支援的功能，跟上游同步時完全不會衝突（`custom/` 從未存在於官方 repo） |
| 覆蓋範圍 | 只做 `QGCCorePlugin` 的 `brandImageIndoor`/`brandImageOutdoor` + URL 攔截器，**沒有**比照 `custom-example/` 做 `CustomFirmwarePlugin`/`CustomAutoPilotPlugin`/`CustomOptions`/`paletteOverride` | 這次需求只是換商標圖示，不是換一套飛控/載具邏輯或色盤；`custom-example/` 完整範本會停用 APM/PX4 Plugin Factory、鎖死單一載具類型等，跟現有一般連線功能衝突，故刻意精簡掉 |
| 既有檔案改動 | **零** | `IS_DIRECTORY custom` 自動偵測是唯一的掛載點，跟 [AVIX_Gimbal_工作報告](AVIX_Gimbal_工作報告_2026-07-20.md) 那次還需要動 7 處既有檔案（`CMakeLists.txt`/`QGCApplication.cc`/`qgroundcontrol.qrc` 等）相比，這次改動面更小 |

---

## 2. 原版 QGC 改了什麼

**新增檔案（全部關在 `custom/` 一個目錄，未動既有邏輯）：**

```
custom/
  CMakeLists.txt                        精簡版（無 CustomModule QML widgets、無 FirmwarePlugin/AutoPilotPlugin）
  cmake/CustomOverrides.cmake           只覆蓋三個圖示路徑變數，未動 APM/PX4/APP_NAME 等其他選項
  custom.qrc                            /Custom/res 前綴，供 URL 攔截器比對
  src/CustomPlugin.h/.cc                brandImageIndoor/Outdoor + URL 攔截器
  res/
    QGCLogoFull.svg / QGCLogoWhite.svg / QGCLogoBlack.svg   raster-in-svg（內嵌 base64 PNG，來源本身非向量圖）
    macx.icns                           16/32/48/128/256/512
    icons/custom_qgroundcontrol.ico     16/24/32/48/64/128/256
    icons/custom_qgroundcontrol.png     256×256（Linux AppImage 用）
  deploy/windows/WindowsQGC.ico         16/24/32/48/64/128/256（EXE 與桌面捷徑圖示）
  android/res/drawable-{ldpi,mdpi,hdpi,xhdpi,xxhdpi,xxxhdpi}/icon.png
```

**修改既有檔案：無。** `custom/` 目錄的存在本身就是唯一的觸發條件，CMake 會自動 `add_subdirectory(custom)`、自動把 `custom.qrc` 併入資源清單、自動把 `CustomPlugin.cc` 併入編譯，全部由官方既有的 `IS_DIRECTORY custom` 判斷式與 `QGC_CUSTOM_BUILD` 巨集完成。

## 3. 圖示覆蓋範圍

| 用途 | 對應檔案 | 覆蓋方式 |
|---|---|---|
| Windows 執行檔／桌面捷徑圖示 | `deploy/windows/WindowsQGC.ico` | `CustomOverrides.cmake` 直接改 `QGC_WINDOWS_ICON_PATH` cache 變數 |
| Windows 執行期視窗／工作列圖示 | `resources/icons/qgroundcontrol.ico` | URL 攔截器：`:/res/qgroundcontrol.ico` → `:/Custom/res/qgroundcontrol.ico`（`QGCApplication.cc:153` 的 `setWindowIcon()` 完全沒改，執行期自動轉向） |
| macOS App Bundle 圖示 | `deploy/macos/macx.icns` | `CustomOverrides.cmake` 改 `QGC_MACOS_ICON_PATH` 指向目錄 |
| Linux AppImage／`.desktop` 圖示 | `resources/icons/qgroundcontrol.png` | `CustomOverrides.cmake` 改 `QGC_APPIMAGE_ICON_PATH` |
| Android Launcher 圖示 | `android/res/drawable-*/icon.png` | CMake 既有的 `custom/android` overlay 機制，逐檔覆蓋到打包用暫存目錄，其餘 `AndroidManifest.xml` 等維持預設 |
| 工具列品牌 logo（已連線時右上角） | `QGCCorePlugin::brandImageIndoor/Outdoor` | `CustomPlugin` 直接回傳 `/Custom/res/QGCLogoFull.svg` |
| App 選單按鈕／關於畫面／Settings 頁面圖示／地圖上無航向資料時的 GCS 標記 | `resources/QGCLogoFull.svg`／`QGCLogoWhite.svg`／`QGCLogoBlack.svg` | 同一組 URL 攔截器，涵蓋 `FlyViewToolBar.qml`、`MainWindow.qml`、`SettingsPagesModel.qml`、`FlightMap.qml` 既有寫死的引用 |

**刻意排除、非缺口：**

| 項目 | 原因 |
|---|---|
| `resources/QGCLogoArrow.svg` | 地圖上「已知航向」時的 GCS 位置標記，會依實際航向旋轉顯示方向。這是功能性方向指示圖示，不是靜態商標；換成目前這個圓形徽章會讓使用者失去「哪個方向」的視覺判讀，故不覆蓋，維持原箭頭圖案 |
| `deploy/windows/installheader.bmp` | NSIS 安裝程式歡迎畫面橫幅，只在安裝當下出現一次，非日常使用會看到的商標，這次需求範圍未提及，暫不處理 |

## 4. 來源圖檔限制

來源圖檔為使用者提供的單張 PNG，**解析度 447×447**，無主題深/淺色變體：

- 所有小於等於 256px 的輸出（Windows ico 各尺寸、Android 各密度、macOS icns 到 256）直接縮小，畫質無虞
- macOS icns 的 512px 檔案、SVG 包裝用的 512px 版本，屬於**向上放大**（447→512，約 1.15 倍），肉眼差異極小，但若之後要支援 Retina/1024px 顯示或更大尺寸輸出，建議跟使用者索取更高解析度或向量原始檔重新產生，目前先以現有來源產出
- 因為只有一張圖、沒有深色/淺色模式變體，`QGCLogoFull.svg`/`QGCLogoWhite.svg`/`QGCLogoBlack.svg` 三個檔案內容目前完全相同

## 5. 驗證過程

沒有只靠肉眼看檔案列表，而是實際跑了一輪 configure → build → 執行：

1. `cmake configure`（Ninja + MSVC 2022 + Qt 6.8.3，沿用既有 `build/QCP_Debug`）：確認輸出 `Enabling custom build` / `QGC: Adding Custom Plugin`，且 APM/PX4/序列埠等既有選項全部維持原狀（`NO`/`OFF`），沒有被 `custom-example/` 範本的預設值誤帶偏
2. 完整 build 成功（1207/1207 物件檔，含 `CustomPlugin.cc`、`qrc_custom.cpp`，最終 `Linking CXX executable Debug\QGroundControl.exe`）
3. 從編譯產出的 `QGroundControl.exe` 用 `System.Drawing.Icon.ExtractAssociatedIcon` 直接抽取內嵌圖示 → 確認是新商標，證明 `.ico` 置換與 CMake 圖示路徑覆蓋正確
4. 實際啟動編譯出的 `QGroundControl.exe`（需補上 GStreamer/Qt 執行期 PATH，比照 [QGC_5.0_Windows_Build_SOP.md](QGC_5.0_Windows_Build_SOP.md) 步驟 7 的執行期環境變數需求，否則會啟動後立即無聲退出），工作列圖示確認顯示新商標——這一步驗證的是 `CustomOverrideInterceptor` 在**執行期**真的有攔截並轉向 qrc 查找，不只是靜態編譯進二進位檔而已
5. 主視窗畫面因為執行環境（背景 session）限制沒能截圖看到工具列品牌 logo，但因為它與已驗證成功的工作列圖示走同一套攔截機制、同一個 `/Custom/res` qrc 前綴，判斷為同樣有效
